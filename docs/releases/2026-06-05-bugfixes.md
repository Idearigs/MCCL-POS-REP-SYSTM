Bug fixes shipped to production after the 2026-06-01 reliability release, found

Each was branched off `develop`, merged clean, then released `develop ‚Üí main`.

## 1. Cash-up override PIN appeared lost after a browser cache clear

**Reported:** owner cleared the browser cache and the manager **Cash-Up Override
PIN** (set per OWNER/MANAGER in User Management) looked gone ‚Äî concern it wasn't
saved in the database.

**Diagnosis:** the PIN *was* always persisted correctly ‚Äî bcrypt-hashed and
tenant-scoped in `users.cashUpPin` (`user-management.service.ts`), with the
close-shift gate verifying it from the DB (`shifts.service.ts` `verifyCashUpPin`).
Clearing the browser cache never touched it. The real defect was UI-only: the
users API never reported whether a PIN existed, so the "PIN" button looked
identical whether or not one was set. After a cache clear there was no indicator,
so it *looked* lost.

**Fix:**
- Backend ‚Äî users-list + get-user now return a `hasCashUpPin` boolean (the hash
  itself is never sent; login/`me` flows verified not to leak it).
- Frontend ‚Äî the PIN button shows green **`PIN ‚úì`** when set, the dialog shows a
  "stored securely in the database" banner, and the list refreshes after
  save/clear so the badge updates immediately.

**Nature:** code-only ‚Äî the `cashUpPin` column already existed from the cash-up
release; no migration. Note the PIN value is one-way hashed and cannot be
recovered by anyone; if forgotten, set a new one to overwrite it.

## 2. Gift-card sale ‚Üí `POST /sales` 404

**Reported:** adding a gift card to the cart and paying by **card** failed with
`POST /api/v1/sales` 404 (Not Found).

**Cause:** the gift-card cart line was sent as a real product (`productId
"giftcard-<ts>"`) with no non-stock marker, so the backend tried to look it up and
threw `NotFoundException`.

**Fix:** generalised the non-stock line handling ‚Äî a module-level
`isNonStockLine()` recognises both `REPAIR SERVICE` and `GIFT CARD` markers; such
lines skip the product lookup and the `sale_items` FK write. Frontend tags the
gift-card line `GIFT CARD: <name>`. Verified on prod: 404 ‚Üí 201
(`SALE-202606-0010`). (PR #25 ‚Üí #26)

## 3. Settings tab row wrapping to two lines

**Reported:** with 10 settings tabs, the tab strip wrapped onto a second row.

**Cause:** `TabsList` used `grid grid-cols-9` against 10 tabs.

**Fix:** switched to `flex w-full justify-start overflow-x-auto
[&>*]:flex-shrink-0` so all tabs sit on one scrollable row. Verified on prod ‚Äî
single row. (PR #25 ‚Üí #26)

## 4. Z-report polish ‚Äî negative-zero + missing department subtotals

**Cause:** the thermal Z-report printed `-¬£0.00` for zero discount/pay-out values,
and department subtotals weren't populated.

**Fix:** added a `fmtNeg()` helper (prints `¬£0.00`, never `-¬£0.00`) used for Total
Discount and Pay-Outs, and wired `buildDepartmentsFromShiftReport` into both print
callers with the backend `getShiftReport` category select. (PR #23)

---

## 5. Settings page crash ‚Äî `ReferenceError: Label is not defined`

**Cause:** the Cash-Up settings tab used `<Label>` but `SettingsPage.tsx` only
imported `FormLabel`. `tsc` passed (ambient global `Label` type) and the vite
build passed (esbuild doesn't error on undefined identifiers), so it only crashed
at runtime.

**Fix:** added `import { Label } from '@/components/ui/label'`. (PR #22)
